## 1. Git Workflow (Mental Model)

Git operates through **three primary states:**

### States

**Working Directory**
- Where files are created and modified
- Untracked and unstaged changes live here

**Staging Area**
- Where selected changes are prepared for commit
- Intentional selection of what to record

**Repository**
- Where changes are permanently recorded as commits
- Immutable history

### Core Flow

Every meaningful Git action respects this flow:
```
modify â†’ stage â†’ commit
```

**Key Insight:** Git is about intentional state transitions, not arbitrary commands.

---

## 2. Core Commands (Conceptual Purpose)

Each Git command represents a **state transition:**

| Command | Purpose |
|---------|---------|
| `add` | Intentionally select changes for staging |
| `commit` | Create an immutable snapshot |
| `push` | Publish local commits to remote repository |
| `pull` | Fetch and merge remote changes |
| `merge` | Integrate divergent histories |
| `stash` | Temporarily store uncommitted work |
| `pop` | Reapply stashed changes |

**Key Takeaway:**  
Git is fundamentally about **state management, not commands.**

## 6. Merge Conflicts (Reality of Team Work)

Merge conflicts occur when:
- The same lines are modified differently
- Git cannot automatically resolve intent

**Rule:**  
> Conflicts are not errors â€” they are **decisions that require human judgment.**

## 7. Forking vs Cloning

**Clone**  
Creates a local copy of a repository

**Fork**  
Creates an independent copy of another repository at the platform level (e.g., GitHub)

**Insight:** Forking is the foundation of **open-source contribution workflows.**

---

## 8. Cherry-Pick (Precision Tool)

**Cherry-pick** applies a specific commit from one branch to another.

**Warning:**  
This is a powerful but **risky tool**. Improper use can lead to:
- Duplicated history
- Maintenance issues
- Confusion in commit lineage

Use only when necessary and with clear intent.
